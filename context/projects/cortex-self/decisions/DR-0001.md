# DR-0001: Task Cache Freshness in tasks.list Query Handler

Date: 2026-05-27
Status: accepted
Evidence: EXP-069

## Context

The `handleTasksList` query handler (`query/tasks.ts`) calls `taskStore.getAll()` which returns tasks from an in-memory cache initialized on first access. Tasks added directly to TASKS.yaml (external edits) are invisible to running daemon's task queries until something triggers `taskStore.refresh()`.

This was discovered during E2E smoke testing (EXP-069): sacrificial tasks added externally required a "cache warm trick" (mutating a cached task to force refresh) before they appeared in `tasks.list` results.

The issue affects TUI dashboard users who add tasks via YAML editing — their changes won't appear in the dashboard without manual cache warm, which is an unreasonable UX burden.

## Options Considered

### Option A: Add `taskStore.refresh()` to the `tasks.list` query handler

Add `deps.taskStore.refresh()` before `getAll()` in `handleTasksList`, so every query fetches fresh data from disk.

- Pro: Always consistent with on-disk state. No cache warm trick needed.
- Pro: One-line change, simple to test.
- Pro: `getActionable()` already refreshes on every call (dispatcher hot path) — the FS-read cost is already part of normal operation.
- Con: Every `tasks.list` query now reads all TASKS.yaml files from disk (~1-5ms per poll).
- Con: Slightly higher latency on each query handler call.

### Option B: Document as known architectural limitation

Create a DR stating that the in-memory cache is intentionally stale, and external YAML edits require a cache warm trick (e.g., issue a non-op mutation).

- Pro: Zero code change. No performance impact.
- Con: Leaves a UX trap for TUI users.
- Con: Requires maintaining the workaround knowledge.
- Con: Inconsistent with `getActionable()` which refreshes on every call.

### Option C: Modify `TaskRepo.getAll()` to always refresh

Change `getAll()` internally to call `load()` every time, aligning it with `getActionable()`.

- Pro: Fixes the issue for ALL consumers, not just the query handler.
- Con: Changes the `getAll()` contract for every caller — dispatcher, archiver, system CLI — many of which depend on cached read semantics.
- Con: Broader blast radius, harder to verify correctness.

## Decision

**Accept Option A**: Add `taskStore.refresh()` to the `tasks.list` query handler.

Rationale:
1. The FS-read cost is already paid by `getActionable()` on every dispatcher tick, so this doesn't introduce a new overhead category.
2. The fix is minimal (one line) and locally scoped to the query handler — no risk to other callers.
3. TUI users get correct behavior without workarounds.

Option C was rejected because it changes `getAll()`'s contract for all callers, including internal consumers that rely on cached reads. Keeping the fix scoped to the query handler is the safest approach.

## Consequences

- `tasks.list` queries now always reflect on-disk TASKS.yaml state.
- Slightly higher latency per `tasks.list` query handler call (one additional `scanAllTasks()` call).
- The cache warm trick in smoke tests (EXP-069 S4/S5) is no longer needed — future smoke tests can add sacrificial tasks directly.
- No change to mutation handlers — they already called `refresh()` internally.
