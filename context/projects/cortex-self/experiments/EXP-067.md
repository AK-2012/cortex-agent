### Scenario 6 — EADDRINUSE double-bind

**Commit tested:** `5ebf944c`
**Command line invoked:** `node scripts/smoke-tui-scenarios-3to7.mjs`

**Primary observed evidence:**
```
[S6] second WebSocketServer on port 3003 → EADDRINUSE error
[S6] primary daemon TUI gateway alive → connect successful after EADDRINUSE test
[S6] daemon status: Cortex daemon is running. PID: 422393
```
- Second `WebSocketServer` on `127.0.0.1:3003` returns `EADDRINUSE: address already in use`
- This is the same error the `TuiGatewayAdapter.start()` method catches at `tui-gateway.ts:144-152` and handles by setting `_noopOutbound = true` + logging a warning
- Primary daemon's TUI gateway was unaffected — subsequent WS connections succeeded
- Unit test `EADDRINUSE soft-failure — second adapter becomes noop` (test 29/119) validates the code path explicitly

**Pass/Fail: PASS** — EADDRINUSE detected, warning log, primary survives.

### Scenario 7 — AskUserQuestion drive-through

**Commit tested:** `<HEAD>` (see smoke output)
**Command line invoked:** `node --import tsx scripts/smoke-tui-askuser.mjs`

**Primary observed evidence:**
```
  ✓ S1: handshake.ack received
  ✓ S2: session.switched received
  ✓ S3: chat.post received (question card)
  ✓ S4: modal.open received
  ✓ S4a: modal.callbackId is ask_user_question_modal_submit
  ✓ S4b: modal has question fields
  ✓ S5: modal.ack received
  ✓ S5a: modal.ack has no errors
  ✓ S6: ask-user.answered event published
  ✓ S6a: ask-user.answered channel matches conduitId
  ✓ S6b: ask-user.answered has answer content

Result: PASS (11/11 passed)
```

The fixture script (`scripts/smoke-tui-askuser.mjs`) creates an isolated test environment:
1. EventBus + TuiGatewayAdapter (ephemeral port) + interaction handlers + hook-bridge subscribers
2. Real WS client connects, handshakes, receives session
3. EventBus publishes `ask-user.requested` → hook-bridge subscriber creates question group
4. WS client receives `chat.post` with question blocks
5. WS client sends `action.click` (actionId `ask_user_question_open_modal`) → `handleOpenModal` sends `modal.open`
6. WS client receives `modal.open` with correct callbackId and question fields
7. WS client sends `modal.submit` with answer (select "Red") → `handleModalSubmit` publishes `ask-user.answered`
8. Bus event captured, answer content verified

**Note:** The hook-bridge subscriber uses `adapter.postMessage` (not `postInteractive`) for the question card, sending `chat.post` frames. The Answer button is embedded in richBlocks as a visual action block. The test drives the flow by directly sending `action.click` with the computed groupId (format: `${sessionId}:${requestId}`).

**Pass/Fail: PASS** — Full AskUserQuestion drive-through flow works through TUI gateway. See follow-up task 5f51.
