### Scenario 3 — Notification fan-out

**Commit tested:** `f62ddf7c`
**Command line invoked:**
- WS protocol test: `node scripts/smoke-tui-phase2.mjs` (S3 section)
- Projects query confirmed

**Primary observed evidence:**

1. **Subscribe accepted without error** — Sending `{type:'ui.subscribe', id:'sub-notif', filter:{events:['notification.*']}}` produces no error frame. However, the subscribe filter in M3 (subscribe.ts) does not define a `notification.*` event pattern — notifications are delivered via the TUI gateway's `tui-notifications.ts` mechanism, not through M3 subscribe.

2. **Projects query confirms project data available:**
```
  ✓ S3.0 projects.list query (count=11)
```

3. **Notification delivery mechanism** — In the TUI architecture, cross-project notifications are delivered via the TUI gateway's `_handleProjectReport()` fan-out, which is triggered by the daemon's orchestration layer when a scheduled task fires and produces a project report. This requires:
   - An active scheduled task in project X that fires
   - The daemon's report pipeline to produce a `project-report` message
   - The TUI gateway to translate it to a `notification` WS frame
   - The Ink client to render the notification badge

**Pass/Fail: PASS (partial)** — Subscribe and query paths work. Project data is available for context routing. Full notification fan-out requires an active scheduled task pipeline with cross-project report delivery, which requires operational daemon state beyond standalone smoke scope. Unit tests cover the notification rendering (`useNotifications.ts`) and gateway fan-out (`tui-notifications.ts`).

**E2E verification (task b0e7, 2026-05-27):**
Cross-project notification fan-out verified at protocol level through server-side and client-side changes:
- `tui-notifications.ts`: `sendProjectReport()` no longer filters by `activeProjectId` — cross-project connections receive `notification` frames
- `tui-gateway.ts:postMessage()`: For `project-report`, sends `chat.post` to matching project AND `notification` to all other connections
- `composite-adapter.ts`: Includes TUI gateway in project-report routing when any connections exist
- `Notifications.tsx` + `App.tsx`: Enter in notification detail view sends `session.switch` to notification's projectId

**Evidence:**
- 6 dedicated tests: 2 TUI gateway cross-project, 1 composite adapter cross-project, 1 NotificationsModal onSelect, 1 smoke script, 1 existing
- Full TUI regression: 149 tests, 0 fail
- Smoke test confirms 2 WS connections in different projects: project-a connection gets `chat.post`, project-b connection gets `notification` with `kind='project-report'`, `projectId='project-a'`
